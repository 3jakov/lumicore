   - MOBILE_READINESS.md

What counts as a finding:
- bug
- security/auth/session problem
- API/shared-types mismatch
- Prisma/data model issue
- migration/reproducibility issue
- contract problem that can break frontend integration
- docs violation that matters for M1 or future mobile compatibility

What NOT to include:
- style nits
- low-value refactor ideas
- “could be improved” unless it creates concrete risk
- frontend review

Important review question:
Can frontend start M1 auth integration without guessing contracts or working around backend gaps?

Format:
1. Findings first, ordered by severity
2. For each finding:
   - severity: P1 / P2 / P3
   - file path
   - lines if possible
   - issue
   - impact
3. Then give:
   - “Backend ready for frontend M1: yes / mostly / no”
   - “Blocking issues”
   - “Top 3 backend follow-ups before/while frontend starts”

Be strict and pragmatic. If something can break handoff, call it out directly.
